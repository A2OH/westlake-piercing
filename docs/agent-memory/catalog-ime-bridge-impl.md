## ✅ WHAT WORKS (proven on device, evidence docs/engine/V3-CATALOG-IME-BRIDGE/)

So the bridge correctly summons the OHOS keyboard + binds the text listener. Material3 SearchBar expands + EditText focuses (blinking cursor) — evidence 01/02 jpegs + ime_chain_stderr.txt.

## ❌ THE BLOCKER (architectural, below the IME layer — bounded per task)
`[OH_IMEHelper] ShowSoftKeyboard rc=61` then `kikakeyboard window background` ~65ms later -> keyboard never visibly stays. ROOT CAUSE (verified, agent traced inputmethod_imf source): **OHOS IMSA ties keyboard visibility to WMS window focus.** `IdentityCheckerImpl::IsFocused` requires `callingPid == WindowManager::GetFocusWindowInfo().pid_`; `PerUserSession::OnFocused` (peruser_session.cpp:812) calls `RemoveClient`->`UnBindClientWithIme`->IME HideKeyboard whenever WMS focus pid != bound-client pid. **The catalog adapter window does NOT hold WMS focus** — `hidumper -s WindowManagerService -a "-a"`: catalog `io.material.catalog WinId 11 ZOrd 1` (composited on top) but **`Focus window: 6` = `EntryView` (launcher/SceneBoard, pid 1565)**. Attach rc=0 means the catalog was MOMENTARILY focused (or system-token bypassed gate 1), but the focus-monitor immediately tears it down because EntryView owns durable focus. This is the SAME long-documented adapter wall as "MMI displayId:-1 compositing / windows don't fully occlude the launcher / touch foundation-blocked / flaky uinput taps" — adapter app windows render over but never durably acquire OHOS WMS input focus. The adapter DOES call `wmsInterface->RequestFocus(windowId)` at session creation (oh_window_manager_client.cpp:586) but SceneBoard/EntryView re-grabs focus. FIXING durable WMS-focus for adapter windows = deep window-manager-adapter change (WindowSessionAdapter/SceneBoard focus arbitration), out of IME-bridge scope, and may be unwinnable vs SceneBoard. ErrorCode 61 is from the NEWER on-device IMSA (on-box v3.1 enum only goes to ~33); behaviorally = the focus-teardown family.

## ★★ TWO BUGS HIT + FIXED DURING BRINGUP (both critical lessons)
1. **DT_NEEDED libinputmethod_client.z.so in the BRIDGE = appspawn-x PREFORK abort.** First impl linked the bridge directly to libinputmethod_client.z.so. That lib has a load-time INIT_ARRAY + pulls libclang_rt.ubsan_minimal.so; loading it into the appspawn-x prefork process aborts libart: `FATAL appspawn-x [primitive.h:133] Primitive char conversion on invalid type`. FIX = split into liboh_ime_helper.so, dlopen'd LAZILY from the forked app process (never prefork). NEVER make an inputmethod/UBSan-bearing .so a DT_NEEDED of the bridge.
2. **`extern "C"` mangling mismatch.** Defined `register_InputMethodBridge` as `extern "C"` (unmangled) but adapter_bridge.cpp declares `extern int register_InputMethodBridge(JNIEnv*)` (C++-mangled `_Z26...`). Linker left it UND -> `MUSL-LDSO relocating failed: symbol not found ... register_InputMethodBridge` -> bridge .so fails to load at JNI_OnLoad -> child dies at "Initializing adapter layer". FIX = plain `int register_InputMethodBridge` (NOT extern "C"), matching the other register_* fns. ★ Prefork poisoning gotcha: once a crashing bridge loads into the appspawn-x prefork, ALL subsequent forks inherit the damage until REBOOT — so isolation tests MUST reboot between .so swaps (a no-reboot revert keeps crashing and gives false signal).
