OHOS launcher (com.ohos.launcher) shows adapter (Android) apps with a blank/default icon + the raw package name as label. âś… FIXED for io.material.catalog â€” the Material Catalog logo (white square + green circle) + "Material Catalog" label now render (evidence `docs/engine/V3-CATALOG-LAUNCHER-ICON-EVIDENCE/`).

**Root cause:** the adapter registers each Android app as an OHOS bundle whose ability `resourcePath` = `/data/app/el1/bundle/public/<pkg>/entry.hap` â€” but that file is NEVER created (the bundle dir has only `android/base.apk`). So the launcher's resourceManager can't load resources â†’ can't resolve the ability's `iconId`/`labelId` â†’ falls back to the default icon + bundleName label.

**KEY (cost hours): the launcher resolves icons/labels CLIENT-SIDE via resourceManager (getAbilityIcon/getAbilityLabel â†’ iconId/labelId), NOT via `bundleResource.db`.** Proven: editing `bundleResource.db` (the BundleResourceManager cache at `/data/service/el1/public/bms/bundle_resources/bundleResource.db`; schema NAME/UPDATE_TIME/LABEL/ICON[data-uri]/SYSTEM_STATE/FOREGROUND/BACKGROUND) had ZERO launcher effect â€” even changing a NATIVE app's LABEL there (TESTCALC123 test) didn't show. Native apps show icons because they HAVE resources (resourcemanager resolves them), independent of bundleResource.db. The launcher also caches resolved label+icon in its layout DB `/data/app/el1/100/database/com.ohos.launcher/phone_launcher/rdb/Launcher.db` (DESKTOPAPPLICATIONINFO: app_name text + appIcon_id ref, NO icon blob â€” icon resolved live). `.db-compare` is just stored CREATE-TABLE DDL (schema check), not a data checksum. Deployed libbms is NEWER than $HOME/foundation source (DB has FOREGROUND/BACKGROUND cols the source's rdb writer lacks) â†’ DON'T rebuild libbms from that source.

**FIX (per-app; no system-component patch; deploy direct â†’ no bm install â†’ no signing):**
1. App's ids: `bm dump -n <pkg> | grep -iE '"iconId"|"labelId"'` (catalog: iconId 16777221=0x01000005, labelId 16777219=0x01000003).
2. Build minimal entry.hap with restool (`$HOME/openharmony/out/sdk/ohos-sdk/linux/toolchains/restool` v4.105): module `src/` = `module.json` (filename MUST be `module.json` â†’ triggers stage-model), `resources/base/media/app_icon.png` (logo PNG), `resources/base/element/string.json` (app_name="<Name>"); + `id_defined.json` `{"record":[{"id":"0x01000005","name":"app_icon","type":"media"},{"id":"0x01000003","name":"app_name","type":"string"}]}` forcing mediaâ†’iconId, stringâ†’labelId. Cmd: `restool -i src -j src/module.json -p <pkg> -o out -r out/ResourceTable.h --defined-ids id_defined.json -f`; verify `out/ResourceTable.txt`.
3. Zip `out/{module.json,resources.index,resources/}` â†’ entry.hap (resources.index + resources/base/media/app_icon.png at zip root).
4. Deploy: push to `/data/app/el1/bundle/public/<pkg>/entry.hap`; `chown installs:installs`; `chmod 644`; `chcon u:object_r:data_app_el1_file:s0`.
5. `rm /data/app/el1/100/database/com.ohos.launcher/phone_launcher/rdb/Launcher.db*` (clears frozen layout) â†’ reboot (or restart com.ohos.launcher). Launcher rebuilds, resourceManager loads entry.hap â†’ real icon + label.
