When making Westlake render an Android app, do NOT add per-app hardcoded branches like `if (pkg.startsWith("com.github.ashutoshgngwr."))`, per-app resource maps generated by `aapt2 dump`, or per-app layout-id direct-inflate shortcuts. These produce visible-progress snapshots but are architecturally wrong: they don't generalize, they pile up, and they hide the real Android API gaps.

**Why:** the user explicitly said "remember don't do per app hack, only do architectural mapping and shim." Earlier in the same session I made noice's Library tab visible by stacking hacks (NoiceResourceMap, NoiceStringMap, hardcoded `home_fragment` inflate when fragmentClass==NavHostFragment && viewId==0x7f090167, hardcoded BNV menu walk, hardcoded "Library" overlay TextView, McD/noice branding in fallback, etc.). Each was a visible win but the cumulative result is an app-specific demo, not a Westlake feature.

**How to apply:**
- When something doesn't work for app X, ask "which Android API surface is incomplete?" and fix that surface generically. Examples: arsc parser missing entries → fix the parser; `inflateMenu` reflection not running → make it apply for any view with `app:menu`; FCV.addView throws → add the bypass for any Fragment-typed container; ConstraintLayout solver missing → set the LayoutParams fields generically by name.
- Per-app constants belong in the APK or in user-supplied config, never in the shim source.
- `NoiceResourceMap`, `NoiceStringMap`, `if (isNoice) { inflate(0x7f0c0042) }`, package-name-keyed branches in branding fallbacks, and similar are anti-patterns. Delete them and rely on the real Android API path.
- If a fix can't be made generic in one session, document the gap and stop — don't ship the per-app shortcut.
