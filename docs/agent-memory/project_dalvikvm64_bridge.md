## Current State (2026-04-12)
- **Dynamic binary** (`dalvikvm-dyn`): 17MB PIE ARM64, uses system linker64, CAN load system .so files
- **Static binary** (`dalvikvm`): 25MB fully static, uses stub JNI libraries
- Both boot with boot image (49K method entry points fixed, 913 native JNI entries cleared)
- Framework classes **through Activity and ContextImpl** initialize on Westlake interpreter
- Instrumentation/ActivityThread fail with UnsatisfiedLinkError (need libandroid_runtime.so natives)

## Key Architecture
- ART v114 from AOSP android-15.0.0_r10
- Boot image: core-oj-a15.jar + core-libart-a15.jar (generated by our dex2oat, CMS GC)
- ALL methods routed through EnterInterpreterFromInvoke (no AOT/JIT)
- InterpreterJni: 30+ shorty patterns for native method C dispatch
- Boot image entry point fixup: VisitPackedArtMethods resets all to interpreter bridges
- JNI stubs: icu_jni_stub, javacore_stub (linux_writeBytes etc.), openjdk_stub

## Key Fixes (Session 2026-04-12)
1. **Stale JNI entry points**: VisitPackedArtMethods also clears data_ for native methods
2. **ClassLinker trampoline override**: OAT header trampolines → runtime assembly stubs
3. **Direct JNI_OnLoad**: Call JNI_OnLoad_icu/javacore/openjdk directly (bypass LoadNativeLibrary dlsym crash)
4. **Native resolution in InterpreterJni**: Check for dlsym lookup stub before calling native → resolve or throw UnsatisfiedLinkError
