## THE FOUNDATIONAL ROOT CAUSE (W15 true bottom)
**Our substrate cannot turn a null-pointer hardware fault into a catchable Java NPE.** Proven precisely:
- McD's RxJava config thread does `x.method()` with `x==null` → SIGSEGV@0 in `nterp_op_invoke_virtual`.
- libart instrumentation (`[W15-ACT]` in `fault_handler_arm.cc::NullPointerHandler::Action`) shows: `ivm=1` (IsValidMethod PASSES), Action computes return_pc, sets `mc->arm_pc = art_quick_throw_null_pointer_exception_from_signal`, returns true. **It does everything right.**
- BUT the crash is at the SAME pc Action redirected FROM (`nterp_op_invoke_virtual+0x2c`); the redirect TARGET (`art_quick_throw_null_pointer_exception_from_signal`) NEVER ran. Only 1 Action call total (no refault loop).
- **Conclusion: OHOS musl's signal chain (MUSL-SIGCHAIN / `signal_chain_handler`) does not propagate ART's modified `ucontext` (arm_pc/cpsr) back to sigreturn → the exception redirect is silently dropped → execution resumes at the faulting instruction → fatal.**
- This breaks ALL implicit runtime exceptions (NPE, implicit array/stackoverflow checks) that ART delivers via signal-context redirect. Latent until an app hits its first real null (McD ran 16s before hitting one). Affects BOTH compiled-frameless (getApplicationInfo: there IsValidMethod also failed) AND nterp paths.

## WHY this is the recurring "wall series"
Every "another null" wall is the SAME defect: stock Android throws a catchable NPE the app absorbs; here every real null is a hard crash. Fixing this one mechanism unblocks McD's splash AND a broad class of apps.

## THE FIX (where to put effort)
Clean systemic fix = make ART's signal-context redirect survive OHOS musl's sigchain (how ART's libsigchain `AddSpecialSignalHandlerFn` integrates with OHOS `signal_chain_handler`'s context save/restore). This is **platform/substrate-level (OHOS musl + [build-host]'s appspawn-x signal setup)** — pair with [build-host], who own the sigchain.
