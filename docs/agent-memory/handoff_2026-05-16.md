## Anti-pattern recognition (V3-specific)

From `V3-ARCHITECTURE.md` §8:

1. **"Just one more shim and we're there"** — if 3rd consecutive CR in
   same integration seam, audit the layer instead of writing CR #4.
2. **"Technical PASS but semantic FAIL"** — CR lands cleanly but visible
   result hasn't moved → suspect test criteria.
3. **"Hand-roll X when AOSP/[build-host] ships a real X"** — real `framework.jar`
   is in [build-host]'s BCP; don't write custom Window / Canvas / View.
4. **"Forking [build-host] to add a method"** — direct edits to `third_party/
   hbc-runtime/` are a one-way door to drift. Either upstream to [build-host],
   shadow the class in our `oh-adapter-runtime-westlake.jar` (PathClass-
   Loader-loaded), or document the blocker and STOP.
5. **"Per-app code branch"** — same rule as V2; same self-audit grep.
6. **"V2-shaped fix in V3 layer"** — porting SoftwareCanvas-style work-
   around into V3 is a misread of the architecture.
