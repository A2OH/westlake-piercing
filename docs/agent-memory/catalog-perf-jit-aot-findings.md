Investigated 2026-06-25 (user asked "is the sluggishness because ART isn't JIT or AOT?"). Answer: **half right — it's an AOT gap, not a JIT problem, and the interpreter isn't even the main cost.**

**Execution mode (proven 3 ways):** framework/boot-image = AOT (dex2oat `speed`, fast). The **catalog APK's own dex = interpreted + JIT** — no app AOT. Evidence: (1) no `oat/arm/base.odex` for any /data/app Android app; (2) `/proc/<pid>/maps` shows `base.apk` mapped `r--s` (raw dex, NON-executable), zero `r-xp` app odex — only `boot*.oat` is r-xp; (3) JIT IS active: `Jit thread pool` thread, `[anon_shmem:dalvik-jit-code-cache]` r-xs, `libart-compiler.so` loaded, libart has 240 JIT symbols, `UseJitCompilation` default true, no `-Xint`. So JIT works; AOT is missing for app dex.

**Where the ~25-30s cold start actually goes (NOT the interpreter):** warm relaunch (prefork hot) → first draw ≈ **3.7s** (measured 3.67-4.31s). The 25-30s is the COLD first-after-boot launch = (a) appspawn-x prefork loading the framework from cold page cache + (b) the custom libart **W-series vtable fixups**: 312 classes relocated / **1014 relocations** across the session (androidx/recyclerview/fragment, com.google.android.material, jacoco — lazy class loads), each at class-LINK time regardless of compiler filter. **AOT would NOT remove (a) or (b).**

**★ Naive AOT is a DEAD END on this device.** AOT-compiling base.apk with stock `$HOME/tools/dex2oat64 --compiler-filter=speed` against the deployed boot image: runtime ACCEPTS it (checksum ok, `K_initialize_oat_file_data=1`) but **crashes deterministically — SIGBUS BUS_ADRALN @0x9 in AOT-compiled jacoco `RuntimeData.getProbes` (aget-object), right after the W9/FIX-VTABLE-A relocations.** Root cause: the deployed libart REWRITES vtable slots at runtime (8 W-fixup markers), but stock dex2oat (0 W-markers) compiled call sites against the STOCK vtable layout → hard-coded vtable indices wrong after relocation → garbage ref → SIGBUS. 312 UI classes get relocated → broad, not excludable. **`speed` AOT only works if dex2oat is rebuilt from the same W-fixup libart source** (build harness only builds libart.so, not the 203MB dex2oat — major effort). The safe fallback `--compiler-filter=verify` (107KB odex, no compiled code → no vtable mismatch → no crash) deploys cleanly but only ~6% warm-start gain (noise).

**Cheap safe win — ✅ APPLIED + VERIFIED 2026-06-25 (libart `ba40f173`, was 2813065e):** gated the FIX-VTABLE-A reloc/summary logging (constexpr `kLogVtableFixup=false`, class_linker.cc ~9130/9140) AND commented the 62 single-line `[*_CP]` checkpoint logs (`// [PERF-2026-06-25 gated]`, all in class_linker.cc). Rebuilt via build_libart_pathA.sh, deployed to /system/android/lib/libart.so (chcon system_lib_file), one reboot. RESULT: per-launch child stderr **43,000 → ~800 lines** (~98% fewer synchronous writes), `[FIX-VTABLE-A-RELOC]`=0, catalog still `drew=1` + grid renders + 0 crash (logic untouched). Old libart backed up /data/local/tmp/libart.pre-perftrim-2813065e; source backup class_linker.cc.bak-preperf. NOTE: wall-clock gain is MODEST (cold-start dominated by prefork/cold-cache + the fixup work itself, not the logging); the real cold-start lever is a WARM appspawn-x zygote (not done). 6 residual `[*_CP]` are from OTHER files (negligible). Original recommendation was: remove/gate the per-relocation `fflush(stderr)` in `$HOME/libart-pathA-work/src/class_linker.cc` (~lines 9136/9143) — that's **1014 synchronous blocking file writes during class loading**. Gate behind a debug flag, rebuild via `build_libart_pathA.sh` (~2min), redeploy /system/android/lib/libart.so. Zero functional risk; cuts real I/O from class load. (Distinct from the deep proxy/vtable fixes which must stay.)
