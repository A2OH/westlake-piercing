**2026-06-03: D-pad/key interactivity for noice on OHOS — root-caused, fix written, but blocked by runtime base mismatch.**

## The full input chain (and where it breaks)
1. OH MMI routes KEY events to the focused window → reach the bridge's `OHInputBridge::OnInputEvent(KeyEvent)` (PROVEN: `[alog:OH_InputBridge] OnInputEvent(KeyEvent): ohCode=2013 -> android=20 ...`). KEY events route fine even though TOUCH is dead system-wide (MMI has no display; keys need no display geometry).
2. Bridge translates OH→Android keycodes (`ohKeyCodeToAndroid`: 2013→20 DPAD_DOWN, 2016→23 CENTER, 2→4 BACK, …) and writes a bit-exact AOSP-11 `type=KEY` InputMessage (104B = 8 header + 96 Key body, enum `Type{KEY=0,MOTION=1}`) to the registered server InputChannel fd via `writeKeyEvent`. PROVEN working: `injectKeyEvent: session=11, action=0, keyCode=20, result=0`. **This is the SEND side, in `liboh_adapter_bridge.so` (md5 2ab08b85, deployed).** Last session built this locally; it has NO regression.
3. **THE GAP — RECEIVE side:** `android_view_InputEventReceiver.cpp` (the InputChannel client-side consumer worker thread, which polls the client fd and dispatches Java events on the main looper via `adapter.window.InputEventBridge.dispatchOnMainThread`) had `case INPUT_MSG_TYPE_KEY:` as a **STUB** — logged `"type=%u seq=%u (not yet dispatched)"` and dropped it. Only `case INPUT_MSG_TYPE_MOTION` built a Java event. So keys were SENT (result=0, fd write OK) but the consumer READ them and threw them away → noice never saw a key. Symptom: result=0 yet NO focus movement and **BACK is a no-op** (BACK is focus-independent — if it reached ViewRootImpl, onBackPressed would fire; it didn't).

## The fix (written + compiles clean, 2026-06-03)
In `$HOME/bridge-build/framework/android-runtime/src/android_view_InputEventReceiver.cpp`: added `WireKeyBody` (96B mirror of bridge's KeyEventBody), cached `KeyEvent` class + the stable 10-arg ctor `(JJIIIIIIII)V` in `JavaRefs`/`ensureJavaRefs`, added `dispatchKeyFromWorker` (mirrors `dispatchMotionFromWorker`: builds `new KeyEvent(downMs,evtMs,action,keyCode,repeat,meta,deviceId,scan,flags,source=0x101 KEYBOARD)` and posts via `dispatchOnMainThread`), and wired `case INPUT_MSG_TYPE_KEY:` to parse `WireKeyBody` + call it. Compiles OK (20936B .o). This file lives in **liboh_android_runtime.so**, NOT the bridge.

noice "Share with friends" = `startActivity(Intent.createChooser(ACTION_SEND text/plain, title))`. createChooser wraps the real SEND intent as EXTRA_INTENT inside an **ACTION_CHOOSER** intent. The adapter `IntentWantConverter.mapAction()` (`$HOME/bridge-build/src/framework/activity/java/IntentWantConverter.java`) maps ACTION_MAIN/VIEW/SEND/PICK but has NO ACTION_CHOOSER case → falls to `default: "ohos.want.action." + androidAction` = bogus `ohos.want.action.android.intent.action.CHOOSER` → OH IAbilityManager.StartAbility(bogus Want) fails → crash. **FIX (universal, one file):** in the intent→Want convert path, if action==ACTION_CHOOSER, replace the intent with `intent.getParcelableExtra(Intent.EXTRA_INTENT)` (the real ACTION_SEND intent, with its own type/EXTRA_TEXT) BEFORE converting → maps to ohos.want.action.sendData → routes to OHOS share. (Caveat: OHOS must have an ability handling sendData; else it's a graceful no-share instead of a crash.) Build = adapter activity-framework jar + deploy + reboot. Other share callers: PresetsFragment (ACTION_SEND text/plain createChooser). Confirmed adapter DOES handle plain ACTION_SEND→sendData; only the chooser wrapper is unhandled.
